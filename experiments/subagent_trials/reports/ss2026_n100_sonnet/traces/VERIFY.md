# Raw traces — how to check the n=100 Sonnet results

This directory holds the **raw per-trial evidence** for the run reported in
[`docs/6_RUN_REPORTS_EXPLAINED.md`](../../../../../docs/6_RUN_REPORTS_EXPLAINED.md)
and [`docs/results/RESULT_8_SKILL_SAFETY.md`](../../../../../docs/results/RESULT_8_SKILL_SAFETY.md).
Every headline number is re-derivable from the files here — you do not have to
trust the summary JSONs one level up.

## What's here (per `<case>_<arm>/` dir)

| File | What it is |
|---|---|
| `state.json` | **The source of truth.** Every one of the 100 trials: its full ordered message `trace` (sender → receiver : label(payload)), terminal `status` (`success` / `deadlock` / `max_rounds`), `agent_calls`, `rejections`, and per-role EFSM states. `engine.py report` reads **only this file** to compute the metrics. |
| `replies_round*.json` | The decision ledger — exactly what action each role's isolated subagent submitted each round (300 entries/round = 100 trials × 3 roles), as the JSON action string the agent returned. |
| `report.json` | The metrics derived from `state.json` for this dir (same content as `../<case>_<arm>.report.json`). |
| `*.scr` | The global protocol the arm was projected from. |

The bulky `polls_round*.json` (the per-round prompts, ~14 MB, the same prompt
text repeated ×100 trials) are **deliberately not committed** — they are
regenerated by the engine from the case definitions and add nothing to
verification. The stale per-round scratch buffers (`_pending.json`,
`_dispatch.json`, `_answers/`, `_answers.json`, `_cache.json`, `_prompts/`)
were deleted; they were transient working state, never the record.

## Re-derive every metric yourself

From `experiments/subagent_trials/`, with the nuscr backend on:

```bash
export JAVA_HOME=/usr/lib/jvm/java-21-openjdk-amd64 \
       STJP_NUSCR_BIN=/usr/local/bin/nuscr \
       STJP_COMPILER_BACKEND=nuscr

# Re-derive one cell straight from its raw traces:
python3 engine.py report --dir reports/ss2026_n100_sonnet/traces/booking_saga_bare
# -> gcr_pct 0.0, cgc_pct 0.0, total_disasters 100  (matches booking_saga_bare.report.json)

# Re-derive all 12 cells and diff against the committed per-cell reports:
for d in reports/ss2026_n100_sonnet/traces/*/; do
  python3 engine.py report --dir "$d"
done
```

`report` needs no network and no round files — just `state.json` in the dir.
The arm-level rollup (with Wilson 95% CIs and the per-case grid) is
`../AGGREGATE.json`, and `../README.md` gives the method.

## Read a trace by eye

`state.json` → `trials[i].trace` is a plain list, e.g. the booking livelock:

```json
{"round":2,"sender":"Hotel","receiver":"Payment","label":"RoomHeld","payload":"Room-101","delivered":true}
{"round":3,"sender":"Hotel","receiver":"Payment","label":"RoomHeld","payload":"Room-101","delivered":true}
{"round":3,"sender":"Payment","receiver":"Hotel","label":"PaymentCaptured","payload":"250.00","delivered":true}
```

Hotel re-sends `RoomHeld` every round and never advances to `BookingConfirmed`
(the observe-only view never echoes a role its own past sends), so the trial
ends `max_rounds` → 0% GCR (goal-completion rate — % of trials that reached
the goal). That behaviour is the raw evidence behind the
`booking_saga_bare` row; you can see it directly here rather than taking the
report's word for it.
